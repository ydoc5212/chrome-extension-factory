# Chrome Web Store Submission

## Prerequisites

- **Chrome Web Store developer account** -- [$5 one-time registration fee](https://chrome.google.com/webstore/devconsole/register)
- **Privacy policy URL** -- must be publicly accessible before submission (see [templates/privacy-policy.md](./templates/privacy-policy.md))
- **Extension icons** -- 128x128 minimum (auto-generated by `@wxt-dev/auto-icons` from source)
- **Screenshots** -- at least one at 1280x800 or 640x400

## Pre-flight

Run the ship-mode validator before every submission:

```bash
npm run check:cws:ship
```

This is structural checks (which CI also runs via `npm run check:cws`) plus submission-readiness checks (real name, real description — no factory placeholders). Zero errors is required before you run `npm run zip`.

See [09-cws-best-practices.md](./09-cws-best-practices.md) for the full rule list, the design philosophy behind the two tiers, and the "why" behind each rule — including things the Chrome Web Store enforces but never writes down in one place (like runtime host-permission requests for faster review).

## Manual Submission

### 1. Build the zip

```bash
npm run zip
```

Output: `.output/<name>-<version>-chrome.zip`

### 2. Upload to CWS

1. Go to the [Chrome Web Store Developer Dashboard](https://chrome.google.com/webstore/devconsole)
2. Click "New Item" (or update an existing one)
3. Upload the zip file

### 3. Fill listing fields

Use the templates in `docs/templates/` to prepare:
- [store-listing.md](./templates/store-listing.md) -- name, description, category
- [privacy-policy.md](./templates/privacy-policy.md) -- required privacy policy
- [cws-submission-checklist.md](./templates/cws-submission-checklist.md) -- full pre-flight checklist

### 4. Upload screenshots

- 1280x800 or 640x400 (1-5 required)
- See [05-launch-materials.md](./05-launch-materials.md) for asset specs

### 5. Submit for review

Review typically takes 1-3 business days. You'll get an email when approved or rejected.

## Automated Publishing with `wxt submit`

WXT supports automated submission via environment variables:

```bash
# Set these in your CI environment or .env file
export CHROME_EXTENSION_ID="your-extension-id"
export CHROME_CLIENT_ID="your-oauth-client-id"
export CHROME_CLIENT_SECRET="your-oauth-client-secret"
export CHROME_REFRESH_TOKEN="your-refresh-token"

# Submit
wxt submit
```

To obtain OAuth credentials, follow the [Chrome Web Store API guide](https://developer.chrome.com/docs/webstore/using-api).

## Common Rejection Reasons

| Reason | Fix |
|---|---|
| Missing privacy policy | Add a publicly accessible privacy policy URL |
| Unjustified permissions | Remove permissions you don't use, or explain each one in the privacy practices section |
| Vague single purpose | Write a clear, specific one-sentence purpose description |
| Remote code execution | Remove any `fetch()` + `eval()` patterns, no loading remote scripts |
| Misleading description | Make the listing accurately reflect what the extension does |
| Broken functionality | Test the production build (`npm run build` + load unpacked) before submitting |

**You get one appeal per violation.** Ship clean the first time.

## Updating an Existing Extension

1. Bump version in `package.json`
2. `npm run zip`
3. Go to CWS dashboard > your extension > "Package" tab
4. Upload the new zip
5. Submit for review

See [04-staleness-prevention.md](./04-staleness-prevention.md) for keeping your extension active.
